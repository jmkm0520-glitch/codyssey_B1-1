# 2. 시스템 설계 문서

## 2.1 문제 정의

회의록은 일반적으로 비정형 텍스트 형태로 작성되기 때문에 회의 후 핵심 의사결정 사항, 담당자, 일정, 후속 업무(Action Item)를 빠르게 파악하기 어렵다.

특히 프로젝트 관리 업무에서는 여러 회의가 반복적으로 진행되므로 중요한 내용이 누락되거나 담당자 간 업무 전달이 원활하지 않을 수 있다.

따라서 AI를 활용하여 회의록을 자동 분석하고 핵심 요약, 결정사항, Action Item, 담당자, 일정 정보를 구조화하여 제공하는 시스템을 설계한다.

본 시스템은 대규모 언어 모델(LLM)을 활용하여 회의록 원문을 자동 분석하고 핵심 정보를 구조화된 형태로 제공함으로써 업무 효율성과 정보 전달의 정확성을 향상시키는 것을 목표로 한다.

# 2.14 환각 검증

## 환각 정의

본 과제에서 환각은 다음과 같이 정의한다.

> 사실, 수치, 정책, 일정 등 검증 가능한 정보를 근거 없이 생성하거나 잘못 전달하는 행위

회의록 요약 업무는 실제 업무 기록을 다루기 때문에 정확성이 중요하다. 따라서 회의록에 없는 담당자, 일정, 예산, 정책을 AI가 임의로 생성하면 환각으로 판단한다.

반면 광고 카피나 소설처럼 창의적 생성이 필요한 작업은 정답이 정해져 있지 않으므로 환각으로 보지 않는다. 다만 허구임을 명시하거나 주어진 세계관 규칙을 지켜야 한다.

## 환각 검증 질문 및 결과

| 번호 | 검증 질문 | 기대 정답 또는 판정 기준 | 참고 근거 | 판정 |
|------|------|------|------|------|
| 1 | 모바일 앱 목표 출시일은 언제인가? | 8월 15일 | 회의록 원문: "현재 목표 출시일은 8월 15일입니다." | Pass |
| 2 | QA 테스트는 언제 시작되는가? | 7월 20일 | 회의록 원문: "QA 테스트는 7월 20일부터 시작할 예정입니다." | Pass |
| 3 | QA 테스트 완료 목표일은 언제인가? | 7월 말 | 회의록 원문: "7월 말까지 완료하는 것을 목표로 하고 있습니다." | Pass |
| 4 | 마케팅 예산은 얼마인가? | 최대 3000만 원 | 회의록 원문: "최대 3000만 원까지 가능합니다." | Pass |
| 5 | 서버 비용 절감 방안으로 검토하기로 한 것은 무엇인가? | AWS Reserved Instance 검토 | 회의록 원문: "AWS Reserved Instance를 검토하면 일부 절감이 가능할 것 같습니다." | Pass |
| 6 | 회의록에 개발 인력 수가 명시되어 있는가? | 명시되어 있지 않음 / 확인 필요 | 회의록 원문에 개발 인력 수 없음 | Pass |
| 7 | 회의록에 앱 출시 연도가 명시되어 있는가? | 명시되어 있지 않음 / 확인 필요 | 회의록 원문에는 "8월 15일"만 있음 | Pass |

## 검증 결과 분석

총 7개 질문 중 7개 Pass를 기록하였다.

특히 회의록에 존재하지 않는 개발 인력 수, 출시 연도와 같은 질문에 대해 확인 필요로 처리하도록 설계하였다.

이는 원문에 없는 정보를 생성하지 않는 환각 억제 전략이 정상적으로 동작함을 의미한다.

## Pass 기준

다음 중 하나라도 충족하면 Pass로 판정한다.

1. 정답을 맞히고 간단한 근거를 제시한다.
2. 정보가 부족할 경우 "확인 필요"라고 명시한다.
3. 확인이 필요한 경우 어디서 무엇을 확인해야 하는지 제안한다.

## Fail 기준

다음 중 하나라도 해당하면 Fail로 판정한다.

1. 틀린 사실을 단정적으로 말한다.
2. 회의록에 없는 정보를 생성한다.
3. 불명확한 질문에 확인 질문 없이 임의로 답한다.
4. 근거 없이 수치, 일정, 담당자를 만들어낸다.

# 6. 심층 인터뷰 대비

## Q1. 만약 비용 제약으로 무료 모델만 사용해야 한다면 품질을 유지하기 위해 프롬프트 또는 프로세스를 어떻게 개선할 것인가?

### 답변

무료 모델은 유료 모델보다 추론 능력과 문맥 유지 능력이 제한될 수 있다.

이를 보완하기 위해 프롬프트를 더욱 구체적으로 설계하고 작업을 단계별로 분리한다.

예를 들어 단순히 "회의록을 요약해줘"라고 요청하는 대신 다음과 같은 절차를 적용한다.

1단계: 회의 참석자 식별

2단계: 결정사항 추출

3단계: Action Item 추출

4단계: 담당자 연결

5단계: 일정 정리

6단계: 원문과 비교하여 누락 여부 확인

또한 결과를 생성한 뒤 다음과 같은 Self-Check 규칙을 추가한다.

- 원문에 없는 정보가 포함되었는가?
- 담당자와 일정이 연결되었는가?
- Action Item이 누락되지 않았는가?

이러한 방법을 사용하면 무료 모델에서도 일정 수준 이상의 품질을 유지할 수 있다.

## Q2. 만약 업무 정책(규정, 수치 정보)이 자주 변경된다면 환각을 줄이기 위한 운영 전략은 무엇인가?

### 답변

정책이나 수치 정보가 자주 변경되는 환경에서는 시스템 프롬프트 내부에 정책 내용을 고정적으로 입력하지 않는다.

대신 최신 정책 문서를 입력 데이터로 함께 제공하고 AI가 해당 문서를 우선적으로 참조하도록 설계한다.

추가적으로 다음과 같은 규칙을 적용한다.

- 정책 근거가 없는 경우 "확인 필요"로 표시
- 최신 정책 버전 번호를 함께 입력
- 수치 정보는 원문 그대로 인용
- 정책 문서에 없는 내용은 추론하지 않음

예를 들어 회사의 휴가 규정이 자주 변경된다면 시스템 프롬프트에 규정을 고정하는 것이 아니라 최신 규정 문서를 함께 입력하여 답변하도록 설계한다.

이러한 방식은 오래된 정보를 사용하는 환각을 줄이는 데 효과적이다.

## Q3. 10턴 이상의 대화에서 문맥이 끊기는 문제가 발생한다면 프롬프트 구조를 어떻게 개선할 것인가?

### 답변

장기 대화에서는 이전에 확정된 요구사항을 AI가 잊어버리는 문제가 발생할 수 있다.

이를 방지하기 위해 대화 중 확정된 조건을 명시적으로 유지하는 구조를 사용한다.

예시

현재 확정 조건

- 대상: 경영진
- 출력 형식: 표
- 포함 항목: 핵심 요약, 결정사항, Action Item, 담당자, 일정
- 금지 사항: 추측성 정보 생성 금지

이후 응답을 생성할 때마다 먼저 현재 조건을 확인하고 결과를 작성하도록 설계한다.

또한 조건 변경이 발생하면 변경된 부분만 수정하고 나머지 조건은 유지하도록 규칙을 추가한다.

예시

변경 전

- 출력 형식: 일반 텍스트

변경 후

- 출력 형식: 표

유지 사항

- 출시일
- 담당자
- 일정
- 예산

이와 같은 구조를 적용하면 장기 대화에서도 문맥 일관성을 유지할 수 있다.

## Q4. 왜 환각을 사실 오류 중심으로 정의하였는가?

### 답변

본 과제의 업무 과업은 회의록 요약 및 Action Item 추출이다.

회의록은 실제 업무 기록이기 때문에 담당자, 일정, 예산과 같은 정보가 정확해야 한다.

만약 AI가 존재하지 않는 담당자나 일정을 생성한다면 실제 업무에 심각한 오류를 초래할 수 있다.

따라서 본 과제에서는 환각을 다음과 같이 정의하였다.

> 사실, 수치, 정책, 일정 등 검증 가능한 정보를 근거 없이 생성하거나 잘못 전달하는 행위
> 

정확성이 가장 중요한 업무 과업이기 때문에 환각을 사실 오류 중심으로 정의하였다.

## Q5. 왜 창의적 생성은 환각으로 보지 않았는가?

### 답변

소설, 광고 카피, 스토리텔링과 같은 창의적 생성은 원래 정답이 존재하지 않는 영역이다.

반면 회의록 요약은 실제 업무 내용을 기록하고 전달하는 작업이다.

따라서 창의적 생성과 사실 기반 생성은 평가 기준이 다르다.

본 과제에서는 사실 기반 업무에서 존재하지 않는 정보를 생성하는 경우에만 환각으로 판정하였다.

## Q6. 환각 검증에서 Pass 기준과 Fail 기준은 무엇인가?

### 답변

### Pass 기준

다음 중 하나 이상을 충족하면 Pass로 판정하였다.

1. 질문에 대한 정답을 맞히고 근거를 제시한 경우
2. 정보가 부족할 경우 "확인 필요"라고 명시한 경우
3. 추가 확인이 필요한 경우 확인 방법을 제안한 경우

### Fail 기준

다음 중 하나라도 해당하면 Fail로 판정하였다.

1. 틀린 사실을 단정적으로 답한 경우
2. 회의록에 없는 정보를 생성한 경우
3. 질문의 전제가 불명확한데도 추측하여 답한 경우
4. 근거 없이 수치나 일정을 만들어낸 경우

따라서 정답을 맞히지 못하더라도 불확실성을 명확하게 표시하면 Pass로 인정하였다.

## Q7. 왜 GPT-4o를 최종 모델로 선정하였는가?

### 답변

본 과제의 업무 과업은 회의록 요약 및 Action Item 추출이다.

이 업무에서 가장 중요한 평가 요소는 정확성, 문맥 이해, 구조화 능력, 환각 억제이다.

비교 결과 GPT-4o는

- 회의록 정보를 가장 정확하게 추출하였고
- 담당자와 일정을 명확하게 연결하였으며
- Action Item을 체계적으로 분류하였고
- 원문에 없는 정보를 생성하지 않았다.

반면 다른 모델은 일정 정보 누락 또는 불필요한 정보 추가가 일부 발생하였다.

따라서 실무 활용성과 정확성 측면에서 GPT-4o가 가장 적합한 모델이라고 판단하여 최종 선정하였다.
